# emu 설계 문서 (Design Invariants)

이 문서는 emu의 **불변식(invariants)** — 최초 설계에서 확정됐고 이후 변경이 넘어서는 안 되는
선 — 을 한 곳에 못 박는다. 새 기능이 이 선을 넘는다면 그것은 기능 추가가 아니라 **다른 도구**다.

> 출처: 최초 설계는 첫 커밋 `09fb474` 와 초기 README에 서술됐다. 진행 항목·의사결정 로그는
> [`BACKLOG.md`](BACKLOG.md), 사용자 문서는 [`../README.md`](../README.md) / [`../README.ko.md`](../README.ko.md).

---

## 1. 정체성 (무엇이고, 무엇이 아닌가)

**emu는 IDE 없이 에이전트가 비대화형으로 모는 Flutter 개발/디버그 inner-loop 도구다.**
해자는 "IDE 없이 에이전트가 모는 루프 + 로그·변수 관찰 가능성"이다.

- **이다**: 기동 → 구동 → 검증 루프를 단발성 CLI 호출로 닫는 드라이버.
- **아니다**: DevTools/IDE의 대체재, e2e 프레임워크(`integration_test`/`patrol`/`maestro`)의 경쟁자,
  UI 단언 엔진. 이들과 **다른 층위**이며 그 위에 로그·변수 검증을 더한다.

이 정체성이 우선순위의 1차 기준이다: **"에이전트 루프의 한 번의 호출로 판정을 강화하는가."**

## 2. 아키텍처 불변식

### I1. 엔진 하나, 프런트엔드 둘
로컬 서버가 `flutter run --machine` 세션(엔진)을 **단독 소유**하고, CLI와 웹 대시보드는
그 서버의 HTTP/WebSocket API **클라이언트**다. 엔진을 소유하는 주체는 항상 서버 하나뿐이다.

```
flutter run --machine  ◄─JSON-RPC─►  engine  ──►  server (127.0.0.1:4577)
                                                    ├─ REST + WebSocket API
                                                    ├─ web dashboard  (browser)
                                                    └─ CLI            (terminal / agent)
```

### I2. 데몬 프로토콜로 제어한다 — 키 전송 금지
제어는 Flutter 데몬 프로토콜(`app.restart` 의 `fullRestart` 토글 등)로 수행한다.
**tmux·키 입력 시뮬레이션·터미널 스크래핑을 쓰지 않는다.** 상태는 프로토콜 이벤트로 관측한다.

### I3. 자급형 단일 바이너리
대시보드 자산(`web/`)을 `bundle_web.dart` 로 바이너리에 임베드해 `build.sh` 가 단일 실행 파일을
만든다. 런타임에 외부 자산 서버·CDN·별도 데몬을 요구하지 않는다.

### I4. 무상태 · 단발성 명령 모델
모든 명령은 시작하고, 판정하고, 끝난다. 앱을 **명령 사이에 멈춰 두지 않는다**(그러면 로그·probe가
멎는다). `probe`/`inspect` 조차 스냅샷을 찍고 **즉시 resume**한다. 대화형 `break`/`step` 을
의도적으로 넣지 않은 이유가 이 불변식이다.

### I5. 에이전트 우선 인터페이스
모든 명령에 `--json` 이 있고, 종료코드로 판정한다. `up`/`reload`/`restart` 는 **최종 상태 + 직후
에러**를 한 번에 반환한다(verdict). 사람의 눈·손을 전제하지 않는다.

## 3. 경계 불변식 (emu가 하지 않는 것)

### B1. UI 단언을 하지 않는다
입력(`tap`/`swipe`/`text`)은 좌표·텍스트를 **주입**할 뿐, 위젯 트리를 질의하지 않는다.
"이 버튼이 보이나?"의 **판단은 에이전트**(스크린샷)가, **검증은 `assert`(로그)/`probe`(변수)** 가 한다.
셀렉터 기반 UI 단언이 필요하면 e2e 프레임워크를 쓰고 emu는 그 위에 로그·변수 검증을 더한다.

### B2. Flutter 위젯 한정
입력은 프레임워크 레벨(VM Service 합성 이벤트 / `updateEditingValue`)이라 네이티브 뷰·웹뷰
텍스트 필드엔 닿지 않는다. emu의 대상은 Flutter 앱이므로 이는 타협이 아니라 범위 정의다.

### B3. 좌표계는 물리 픽셀 하나
사용자·에이전트에게 노출하는 좌표는 **물리 픽셀**뿐이다(`shot` 이 캡처하는 공간). 논리 좌표 변환은
백엔드가 dpr로 내부 처리한다. 플랫폼(Android/iOS)이 달라도 좌표 규약은 동일하다.

### B4. 학습 메모리는 권위가 아니다
`.emu/memory.json` 의 학습값(dpr·화면크기 등)은 재계산 재료일 뿐, 다음 실행이 이걸 믿고
스크린샷 없이 탭하지 않는다. **좌표는 부팅마다 썩으므로 저장하지 않는다.** 설정도 "머신·시간을
넘어 참인가"라는 커밋 경계로 계층화한다(참이면 `emu.yaml` commit, 지금만 참이면 `emu.local.yaml`).

## 4. 최초 설계 반영 현황

| 설계 기둥 (첫 커밋 `09fb474`) | 불변식 | 상태 |
|---|---|---|
| 엔진 하나, 프런트엔드 둘 | I1 | ✅ 첫날부터 유지 |
| IDE 없이 Flutter 디버깅 | 정체성 | ✅ 유지 |
| hot reload / restart / cold restart | I2 | ✅ `reload`/`restart`/`cold` |
| 검색·필터 가능한 디버그 로그 | I5 | ✅ `logs` + 대시보드 |
| 데몬 프로토콜 제어, tmux 금지 | I2 | ✅ 유지 |
| 자급형 단일 바이너리 | I3 | ✅ 유지 |
| CLI + 웹 대시보드 공유 | I1 | ✅ 유지 |

**최초 설계에서 제거·타협된 항목은 없다.** 이후 추가된 것은 모두 원설계의 확장이다:

| 확장 | 강화한 불변식 | 상태 |
|---|---|---|
| `assert` — 로그 오라클 | I5, B1 | ✅ |
| `probe` — 변수 오라클 | I4, B1 | ✅ |
| `tap`/`swipe`/`text` — VM Service 입력(Android·iOS, 유니코드) | B1, B2, B3 | ✅ |
| `up` 첫 프레임 대기(verdict) | I5 | ✅ |
| `inspect` — 지역변수+콜스택 스냅샷 | I4 | ✅ |
| 계층 설정(`emu.yaml`) + 학습 메모리 | B4 | ✅ |
| Claude Code 플러그인 패키징 | 정체성(에이전트 우선) | ✅ |
| 인터랙티브 대시보드(클릭=탭) | I1 | ✅ |

보류 항목(전부 근거 있는 보류)도 불변식으로 설명된다:
- 🚫 `emu e2e`(외부 e2e 엔진 구동) — 입력 수단 확보로 전제 소멸(B1과 중복).
- 🚫 대화형 `break`/`step`/`continue` — **I4(무상태·단발성)와 정면충돌**.
- 🚫 비-Flutter 텍스트 입력 — **B2(Flutter 한정)** 밖 + 견고·무의존 경로 없음.

## 5. 변경 판단 기준

새 기능을 검토할 때 이 질문을 통과해야 한다:

1. **I1~I5 중 하나라도 깨는가?** 깨면 그건 emu가 아니라 다른 도구다. 재설계로 취급한다.
2. **B1~B4의 경계를 넘는가?** 넘으면 범위 확장이므로 명시적 근거와 함께 이 문서를 갱신한다.
3. **에이전트 루프의 "한 번의 호출로 판정"을 강화하는가?** 아니면 우선순위가 낮다.
4. **실수요가 있는가, 투기인가?** 투기면 근거를 남기고 보류한다(BACKLOG의 보류 패턴).
